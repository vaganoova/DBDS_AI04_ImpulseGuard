# Datasets

## Overview

The project uses:
- Synthetic transaction dataset (10,000 transactions)
- Real user feedback collected through the app (appended automatically on retrain)

---

## Features

| Feature | Column Name | Type | Description |
|---------|-------------|------|-------------|
| Purchase time | `hour` | 0–23 | Hour of the day when purchase was made |
| Price | `price` | 5.00–500.00 | Purchase amount |
| Product category | `category` | text | clothing, food, electronics, entertainment, home, beauty |
| Purchase frequency | `frequency` | 1–20 | How often the user buys in that category |
| Essential need | `is_essential` | 0 or 1 | 1 = genuine need (e.g. appliance, food), 0 = want |
| Deliberation time | `deliberation_minutes` | 1–20160 | Minutes spent deciding before buying |
| Planned ahead | `on_wishlist` | 0 or 1 | 1 = was on a wishlist / shopping list |
| Label | `impulse_level` | 0–3 | 0=none, 1=mild, 2=moderate, 3=strong |

The label is **graded** (a spectrum), not a binary yes/no, because impulse
buying comes in degrees.

---

## Why These Features?

The key insight is that **impulse buying is defined by lack of deliberation and
necessity, not by price or time of day**:

- **`deliberation_minutes`** — buying within minutes is a strong impulse signal;
  considering an item for days is not. Drawn from overlapping lognormal
  distributions (wishlist items skew longer) so it is realistic and not a
  deterministic function of `on_wishlist`.
- **`on_wishlist`** — planned, pre-committed purchases are not impulsive.
- **`is_essential`** — a genuine need is not an impulse buy even if expensive or
  bought late at night.
- **`category`** — clothing, beauty and entertainment skew emotional / want-driven.
- **`price`** — has a *mild* role for non-essential purchases only; it is the
  weakest predictor (expensive ≠ impulsive on its own).
- **`hour`** — some impulse buys cluster at night (weak signal).
- **`frequency`** — repeated buying in a category can indicate habitual behaviour.

## Dataset Generation

The dataset is generated by `Source/Data_Generation/generate_dataset.py`.
Each purchase gets an `impulse_score` from a rule-based scoring system where
impulse is driven by behaviour, and essential/planned purchases are pulled
*away* from "impulsive":

| Condition | Score |
|-----------|-------|
| Deliberated < 10 minutes | +2 |
| Deliberated 10–60 minutes | +1 |
| Not on a wishlist | +1 |
| On a wishlist (planned) | −2 |
| Essential need | −2 |
| Hour between 22:00–05:00 | +1 |
| Category is clothing, beauty, or entertainment | +1 |
| Frequency > 10 | +1 |
| Non-essential AND price > 300 | +1 |

**Label noise:** Gaussian noise (mean 0, std 1.0) is added to the score before
bucketing. This means identical-looking purchases do not always get the same
label — so the model must *learn* the pattern instead of memorizing a fixed
rule. This is what makes the evaluation metrics meaningful.

The noisy score is then bucketed into a level:

| Noisy score | Level |
|-------------|-------|
| ≤ 0 | 0 — Not impulsive |
| 1–2 | 1 — Mildly impulsive |
| 3–4 | 2 — Moderately impulsive |
| ≥ 5 | 3 — Strongly impulsive |

Note that **price is not a primary driver** of the label (only a mild
non-essential bonus) — this is what fixes the "expensive purchase = impulsive"
mistake.

Approximate balance (10,000 rows): level 0 ≈ 3300, level 1 ≈ 2600,
level 2 ≈ 2250, level 3 ≈ 1900.

## Real User Feedback

When a user corrects an assessment in the app, the purchase and its true level
are appended to `Data/Real_User_Data/feedback.csv`. On the next training run,
`preprocess.load_dataset()` automatically merges this feedback with the
synthetic data, so the model gradually learns from real behaviour.

---

## Limitations

- Synthetic patterns are rule-based (now with noise), not real user behaviour.
- No personalization yet — impulse is relative to each individual.
- Real-world data would contain more noise and edge cases.
